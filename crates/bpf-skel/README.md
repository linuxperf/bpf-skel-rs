# bpf-skel-rs

Rust-facing skeleton library generated from third-party BPF programs.

This crate publishes pre-generated libbpf-rs skeleton modules. The BPF object
bytes are embedded in the generated Rust sources under `src/skel`, so downstream
users can depend on the crate without running the BPF build toolchain.

The repository still includes the generation tooling for maintainers who need to
refresh `crates/bpf-skel/src/skel` from the BPF sources.

## Vendor Layout

BCC is tracked as the default git submodule at:

```text
xtask/third_party/bcc
```

After cloning this repository, initialize it with:

```bash
git submodule update --init --recursive
```

Vendor files are not modified. The generation task walks each direct child
directory under `xtask/bpf/`, invokes its Makefile, scans
`xtask/bpf/<name>/.output` for `*.bpf.o`, then generates Rust skeletons into
`crates/bpf-skel/src/skel` from those objects.

For the default BCC repository, the wrapper Makefile in `xtask/bpf/bcc/`
delegates to `xtask/third_party/bcc/libbpf-tools`.

## Refreshing Skeletons

The published crate does not run skeleton generation during normal
`cargo build`. To regenerate the checked-out skeleton sources, run:

```bash
cargo run -p xtask -- gen-skel
```

This command builds every configured BPF workspace and writes generated Rust
files under:

```text
crates/bpf-skel/src/skel/
```

`crates/bpf-skel/src/skel` is ignored by Git as a generated directory, but it is
explicitly included in the crate package by `crates/bpf-skel/Cargo.toml`.
Release commits must force-add the refreshed skeletons so GitHub Actions can
check them out:

```bash
git add -f crates/bpf-skel/src/skel
```

## Generated Library

Every configured repository gets its own namespace under `bpf_skel::skel`.

For example, BCC `profile.bpf.c` becomes:

```rust
use bpf_skel::skel::bcc::profile;
```

## Adding Repositories

Add a new direct child directory under `xtask/bpf/`, for example
`xtask/bpf/foo`, and put a Makefile in that directory. The Makefile should write
generated BPF objects to its local `.output` directory.

Example:

```text
xtask/bpf/foo/Makefile
xtask/bpf/foo/.output/*.bpf.o
```

The exact builder and skeleton type names are generated by `libbpf-cargo` from
the BPF object.

## Requirements

For using the published crate:

- Rust stable
- Runtime system support required by `libbpf-rs` and the selected eBPF program

For regenerating skeletons:

- Rust stable
- `make`
- `clang`
- `llvm-strip`
- Third-party repositories initialized as needed
- Repository Makefiles that can build `.output/<name>.bpf.o` targets for the
  current host platform

## Packaging

The root crate is configured with `build = false` and packages only the public
library sources:

```text
LICENSE*
README.md
src/**
```

This keeps `xtask/`, local `.output` artifacts, and the generation tooling out
of the published crate while still shipping the pre-generated skeleton modules.

Check the package contents with:

```bash
cargo package -p bpf-skel --list
```

Build and verify the package with:

```bash
cargo package -p bpf-skel
```

## Publishing

Publishing is handled by GitHub Actions. Configure the repository or
`crates-io` environment secret:

```text
CARGO_REGISTRY_TOKEN
```

Then push a version tag:

```bash
git tag v0.1.0
git push origin v0.1.0
```

The publish workflow verifies that `crates/bpf-skel/src/skel` exists, checks the
package contents, and runs `cargo publish --locked` from the crate directory. It
does not regenerate skeletons; refresh `crates/bpf-skel/src/skel` before tagging
a release.
